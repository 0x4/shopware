Shopware Checkliste für neue Backend-Module

Dies ist eine Checkliste, die jeweils durchlaufen werden soll, wenn ein Modul vor der Fertigstellung ist. In dieser Checkliste befinden sich die wichtigsten Punkte, die ein Modul erfüllen muss. Jeder Entwickler ist dafür verantwortlich, die entsprechenden Punkte zu implementieren, um eine Konsistenz modulübergreifend zu erreichen.

Struktur Namespace und Ordnerstruktur

  1. Korrekte Ordnerstruktur verwendet?

    Damit jeder Entwickler das Modul weiterführen / debuggen / fixen kann, sollte folgende Orderstruktur verwendet werden: 

    /
  /controller
    main.js - Beispiel-Controller für die SubApplikation
  /model
  /store
  /view
    /main - Jeder Controller bekommt eigene Views
      list.js
  app.js - Initialisiert die SubApplikation

  2. Namespace korrekt angegeben?

    Wurde der korrekte Namespace angegeben? Dies ist wichtig, damit der Auto-Loader korrekt funktioniert. Unser Prefix bei Namespaces ist "Shopware.". Dahinter folgt der Typ wie z.B. "Shopware.view" und der Name der Komponente wie z.B. "Shopware.view.Viewport". Bei SubApplikationen, so wie jedes Modul sein wird, sieht der Namespace ein wenig anders aus, aber hat den gleichen Aufbau. Gehen wir davon aus dass wir aktuell die Benutzerverwaltung erstellt. Diese hätte als Namespace-Prefix "Shopware.apps.Auth.". 

    // Subapplication init controller
Shopware.apps.Auth

// Subapplication controller - Controllername is "Main"
Shopware.apps.Auth.controller.Main

// Subapplication Store - Storename is "Users"
Shopware.apps.Auth.store.Users

// Subapplication Model - Modelname is "User"
Showpare.apps.Auth.model.User

// Subapplication view - Controllername is "Main"
Shopware.apps.Auth.view.main.List

  3. Wurde ein Smarty-Block zur späteren Anpassbarkeit in jede Modul-Datei hinterlegt?

    Es ist so vorgesehen, dass jeder Plugin-Entwickler später die Möglichkeit haben soll jedes einzelnes Modul anzupassen. Deshalb sollte sich in jeder Modul-Datei ein Smarty-Block hinterlegt werden.

Allgemein

  1. Standard-Layout benutzt?

    Dieser Punkt ist sehr wichtig, da wir durch ein Standard-Layout eine Konsistenz in die Applikation bringen. Durch Konsistenz können wir eine Art "Sicherheit" für den Endkunden schaffen, der das Modul benutzt. Durch diese Art von "Sicherheit" weiß der Endkunde genau, wie er alle Module benutzen kann.

  2. Korrekte Methodenbenennung

    Bitte achtet darauf, dass die Methoden korrekt in Camelcase (erstes Wort bitte "klein" schreiben) geschrieben sind und das alle Eventlistener-Methoden als Prefix "on" haben. 

    onSaveBtn: function(btn, event) {
   // do something
}

  3. Korrektes UI angegeben?

    Als "ui"-Angabe sollte immer "default shopware-ui" gewählt sein. Dadurch haben die Templater und Designer es später einfacher, das Shopware-Design für das jeweilige Modul anzupassen bzw. zu gestalten. 

    Ext.define('Shopware.view.Viewport', {
   ui: 'default shopware-ui'
});

  4. Wurde ein Widget-Name bei Views angegeben?

    Ein Widget- bzw. Alias-Name bringt mehrere Vorteile mit sich. Zum einen besteht dann die Möglichkeit das Modul per Ext.widget() zu öffnen und zum anderen hilft es ungemein bei der Adressierung in Selenium Tests. 

    Ext.define('Shopware.apps.Auth.view.main.List', {
   alias: 'widget.auth-main-list'
})

  5. Resize des Moduls überprüft?

    Kann ein Modul über die "Resize"-Funktion von ExtJS 4 verkleinert bzw. vergrößert werden, ohne dass das Layout zerstört wird. Um dies zu erreichen sollte man das "border"-Layout nutzen.

  6. Dokumentation

    Wurde das Modul komplett "inline" durch kommentiert? Dies ist wichtig damit andere Entwickler nahtlos an dem Modul weiterarbeiten kann. Eine ordentliche Dokumentation erwartet ein "Klassenkommentar", welches die Komponente und deren Zweck beschreibt. Des Weiterem gehört auch über jede Methode ein Kommentar mit den erwarten Parametern und den Rückgabewert. Wenn eigene Attribute verwendet wurden, sollten diese auch kommentiert werden. 

    /**
 * Shopware UI - Viewport
 *
 * This component creates the Shopware
 * viewport. It's doubles the width of the viewport to
 * holds the two desktops containers.
 *
 * @author: s.pohl <stp@shopware.de>
 * @date: 2011-12-05
 */
Ext.define('Shopware.view.Viewport', {
   extends: 'Ext.container.Viewport',
   ui: 'default shopware-ui',

   /** Holds the shopware desktops */
   desktops: [],
	
   /**
    * Initializes the component, binds the necessary event listeners
    * and sets width to the double viewport width.
    */
    initComponent: function() {
    
       this.on('click', this.onViewportClick, this);

       // do something
       this.callParent(this);
    }
    
    /**
     * Event handler which do something if
     * the user clicks on the viewport
     *
     * @param view - Ext.container.Viewport
     */
    onViewportClick: function(view) {
       // do somehting on click
    }
}

  7. Modul komplett getestet?

    Wurde das Modul durch den Entwickler noch einmal komplett mit allen Funktionen und Konfigurationsmöglichkeiten durchgetestet? So können wir als Entwickler die QA entlasten.

  8. Rückmeldung an den Benutzer?

    Wird dem Benutzer eine Rückmeldung über Erfolg oder Misserfolg seiner Aktion gegeben? Diese Rückmeldung kann entweder "Inline" im Modul selbst dargestellt werden oder über eine Ext.MessageBox. Bitte auch darauf achten, dass die Fehlermeldungen aussagekräftig sind. Auf das Fehlverhalten von Benutzern kann hier natürlich eingegangen werden, wichtig ist, dass der Ton stimmt.

  9. Textbausteine verwendet?

    Wurden für alle Texte im Modul Textbausteine verwendet? Dies ist wichtig, da das Shopware Backend später in verschiedenen Sprachen zur Verfügung gestellt werden soll.

  10. Rechtschreibung überprüfen

    Bitte kontrolliert noch einmal Eure Rechtsschreibung. Rechtschreibfehler sind in einer professionellen Software nicht erwünscht.

  11. History-Support

    Wurde der History-Support eingebaut, so dass der Benutzer auch über die Browser-Buttons navigieren kann? Weitere Informationen findet man in der ExtJS 4 Dokumentation unter dem Punkt Ext.util.History 

    // Handle tab changes on a TabPanel
tabPanel.on('tabchange', function(tabPanel, tab){
     Ext.History.add(tabPanel.id + ':' + tab.id);
});

  12. Ist das Modul "stateful"?

    In Shopware bieten wir einen State-Support an. Dies bedeutet, dass sich die Module immer an der gleichen Stelle und Größe öffnen. Dazu müssen in der Modulkonfiguration die beiden Attribute "stateful" und "stateId" gesetzt sein. Weitere Informationen findet man in der ExtJS 4 Dokumentation. 

    Ext.define('Shopware.apps.Auth.view.List', {
   stateful: true,
   stateId: 'shopware-auth-list'
});

  13. Werden die Events im Controller gesetzt?

    Um das MVC (Model-View-Controller) Prinzip korrekt umzusetzen, sollten alle Event-Listener im Controller des Moduls gesetzt werden. ExtJS 4 bietet zwar die Möglichkeit die Events in der View selbst zu setzen, dies ist aber nicht empfehlenswert. 

    Ext.define('Shopware.apps.Auth.controller.Main', {
   extends: 'Ext.app.Controller',
	
   init: function() {
      this.controll({
         'grid': {
            render: this.onGridRender
         },
         'grid button[action=add]': {
            click: this.onAddUser
         }
      })
   }
});

  14. Können mehrere Fenster desselben Moduls geöffnet werden?

    Bitte prüft auch, ob ein Modul in mehreren Fenstern geöffnet werden kann. Sprich können mehrere Instanzen erstellt werden? Hier kann es zu Fehlern z.B. bei den dockedItems kommen. 

    initComponent: function() {
   //.. some logic
   this.dockedItems = Ext.clone(this.dockedItems);
   this.callParent(this);
}

  15. Wurde der Ext.FocusManager verwendet?

    Der Ext.FocusManager bietet die Möglichkeit eine grundlegende Keyboard-Navigation ohne weitere Programmieraufwand zu implementieren. Der Ext.FocusManager ist standardmäßig aktiviert und muss in diesen Punkt nur geprüft werden.

  16. Kann das Fenster minimiert und maximiert werden?

    Damit der Fenstermanager korrekt funktioniert und um den Benutzer ein Desktop-ähnliches Verhalten zu bieten, sollten alle Fenster zumindest minimiert und maximiert werden können. 

    Ext.create('Ext.window.Window', {
   layout: 'border',
   width: 800,
   maximizable: true,
   minimizable: true,
   stateful: true,
   stateId: 'shopware-auth-list',
   title: 'Benutzerverwaltung',
   items: [ this.getView('main.List') ]
}).show();

Formulare und Formular-Elemente

  1. Sind alle Labels "bold" und "rechtsbündig"?

    Um einen konsistenten Look'n'Feel zu bekommen, sollten alle Labels rechtsbündig und fett sein. Wichtig ist hier auch, dass alle Labels im Modul eine einheitliche Länge haben (z.B. 150px).

  2. Hilfs- und Placeholdertexte gesetzt?

    Dieser Punkt hilft die Usability unserer Backend-Module zu steigern. Durch Placeholdertexte geben wir ein Inline-Beispiel im Feld an. Hilfstexte können entweder unter ein Formular-Element gesetzt oder über ein Hilfe-Icon am Ende des Formular-Elementes zur Verfügung gestellt werden. 

    this.nameField = Ext.create('Ext.form.field.Text', {
   fieldLabel: 'Name',
   allowBlank: true,
   supportText: 'Lorem ipsum dolor'
   helpText: 'Lorem ipsum dolor'
});

  3. Findet eine Validierung statt?

    Um dem Benutzer die Angabe von Daten zu Erleichtern sollten immer alle eingegeben Daten validiert werden. ExtJS 4 bietet z.B. eine Validierung über das Ext.data.Model an. Weitere Informationen findet man in der ExtJS 4 Dokumentation. 

    Ext.define('User', {
   extend: 'Ext.data.Model',
   fields: [
        {name: 'name',     type: 'string'},
        {name: 'age',      type: 'int'},
        {name: 'gender',   type: 'string'}
    ],
    validations: [
        {type: 'presence',  field: 'age'},
        {type: 'length',    field: 'name',     min: 2},
        {type: 'inclusion', field: 'gender',   list: ['Male', 'Female']}
    ]
});

    Jetzt hat man die Möglichkeit über das Ext.form.Panel die Validierung per form.isValid() anstoßen und zu prüfen

    this.submitBtn = Ext.create('Ext.button.Button', {
   text: 'Speichern',
   formBind: true,     // wird nur einmal aktiv, wenn das Formular valide ist
   handler: function() {
      var form = this.up('form').getForm();
      
      if(form.isValid()) {
         form.submit();
      } else {
         // Zeige Fehlermeldung an
      }
   }
});

  4. Gewichtung der Buttons

    Wurde die Gewichtung der Buttons eingehalten? Der "Speichern"-Button sollte eine kontrastreiche Farbe erhalten und größer sein als der "Abbrechen"-Button. Der "Speichern"-Button befindet sich immer rechts neben den "Abbrechen"-Button. 

    this.buttons = [{
   text: 'Abbrechen'
}, {
   text: 'Speichern',
   action: 'save',
   scale: 'large',      // "scale" beeinflusst die Größe des Buttons
   cls: 'primary'       // erhählt durch die Klasse den Kontrast
}];

  5. Beschriftung der Buttons

    Wurden die Buttons nach Sinn und Zweck beschriftet? Es ist wichtig, modulspezifische Buttontexte zu verwenden. So bekommt der Button einen besseren Bezug zum Formular und der Benutzer weiß genau was passiert, wenn er einen bestimmten Button betätigt.

  6. Wurden so wenig Formular-Elemente wie möglich verwendet?

    Um das Backend so einfach wie Möglich zu halten, sollten wir auf überflüssige Formular-Elemente verzichten. So können wir die Usability unserer Applikation steigern und den Workflow des Benutzers optimieren.

Stores und Models

  1. Felderdeklaration korrekt?

    Wurde bei der Felderdeklaration der richte Typ angegeben? Wurde bei einem Datumsfeld das richte Datumsformat hinterlegt? 

    Ext.define('User', {
   extend: 'Ext.data.Model',
   fields: [
      {name: 'name',     type: 'string'},
      {name: 'age',      type: 'int'},
      {name: 'alive',    type: 'boolean', defaultValue: true}
   ],
});

  2. storeId hinterlegt?

    Um den Ext.data.StoreManager zu verwenden, um Stores z.B. für die komplette Applikation zur Verfügung zu stellen muss eine storeId hinterlegt werden. 

    Ext.create('Ext.data.Store', {
   model: 'User',
   storeId: 'usersStore'
});

  3. pageSize hinterlegt?

    Eine pageSize sollte bei jedem Store hinterlegt sein, um ggf. später ein Paging einzubauen. 

    var store = Ext.create('Ext.data.Store', {
   model: 'User',
   pageSize: 10,
   storeId: 'usersStore'
});
store.load({ params: { start: 0, limit: 10 } });

  4. Datenverarbeitung komplett Store basierend?

    Ist der komplette Datenaustausch mit dem serverseitigen Backend komplett auf Stores aufgebaut? ExtJS 4 bietet ein hervorrangendes System für die Datenverarbeitung - dieses sollte auch genutzt werden. 

    Ext.define('Shopware.apps.Auth.model.Main', {
   extend: 'Ext.data.Model',
   fields: [ /** .. some fields */ ],
   proxy: {
      type: 'ajax',
      api: {
         read: 'backend/auth/getUsers',
         create: 'backend/auth/createUser',
         update: 'backend/auth/updateUser',
         destroy: 'backend/auth/deleteUser'
      },
      reader: { type: 'json', root: 'data' }
   }
});

Grids und deren Toolbars

  1. Wurde eine Paging-Toolbar eingebaut?

    Eine Paging-Toolbar bietet mehrere Vorteile. Es findet eine Performance-Steigerung statt, da nicht alle Datensätze auf einmal angezeigt werden und der Benutzer erfährt direkt wie viele Einträge insgesamt zur Verfügung stehen. 

    Ext.create('Ext.grid.Panel', {
   store: store,
   columns: [
       { header: 'Name',  dataIndex: 'name' },
       { header: 'Email', dataIndex: 'email', flex: 1 },
       { header: 'Phone', dataIndex: 'phone' }
   ],
   dockedItems: [{
       xtype: 'pagingtoolbar',
       store: store,   // gleicher Store wie beim GridPanel
       dock: 'bottom',
       displayInfo: true
   }]
});

  2. Steht eine Suche zur Verfügung?

    Steht dem Benutzer eine Suche zur Verfügung, um schneller navigieren zu können? Die Suche sollte bei großen Datensätzen "Remote" über den Ext.data.Store laufen.

  3. Können Spalten sortiert werden?

    Standardmäßig sind alle Spalten in ExtJS 4 sortierbar. Bei großen Datensätzen sollte das Sortieren "Remote" über den Ext.data.Store laufen.

  4. Action-Buttons im Grid

    ExtJS 4 bietet für diesen Fall eine eigene Spalte namens "actioncolumn" an, womit man Action-Buttons anhand von einen Icon und einen Hilfetext erstellen kann. 

    this.columns = [{
   xtype: 'actioncolumn',
   width: 50,
   items: [{
      iconCls: 'delete',
      action: 'delete',
      cls: 'delete',
      tooltip: 'Löscht diesen Benutzer',
   }, {
      iconCls: 'pencil',
      cls: 'editBtn',
      tooltip: 'Bearbeiten Sie diesen Benutzer'
   }]
}];
